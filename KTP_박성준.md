# Cursor AI를 활용한 AI Agent 코드 리팩토링 KPT 회고

## 1. 코드 탐색

### Keep

- 처음 보는 레거시 코드에 진입할 때 AI가 클래스 다이어그램, 함수 호출 구조, UI와 내부 로직의 연결 관계를 빠르게 파악해 주는 점이 좋았다.
- 실제 UI 동작과 함수 호출 흐름을 바탕으로 시뮬레이션 상태를 그림이나 구조로 정리해 주어 초기 이해 비용을 줄일 수 있었다.
- 사람이 직접 코드를 모두 읽기 전에 전체 구조를 먼저 잡아볼 수 있어 리팩토링의 출발점이 명확해졌다.

### Problem

- AI는 사람보다 프롬프트에 훨씬 민감하게 반응했다.
- 사람은 대화 속에 틀린 단어나 문장이 있어도 분위기와 맥락을 보고 실수로 보정하지만, AI는 잘못 들어간 문장도 그대로 수행하려는 경향이 있었다.
- 예를 들어 Green 단계에서 평가만 진행하고 있었는데, 프롬프트에 실수로 `Git에 제출하라`는 문장이 포함되면 실제 제출까지 진행하려고 했다.

### Try

- 코드 탐색 단계에서는 AI에게 수행 범위를 명확히 제한하는 프롬프트가 필요하다.
- `분석만 수행`, `파일 수정 금지`, `Git 명령 실행 금지`처럼 금지 조건과 허용 조건을 명확히 작성해야 한다.
- AI가 분석한 코드 구조가 실제 코드와 일치하는지 검증할 수 있도록 주요 파일, 함수, 호출 관계를 함께 기록하는 절차가 필요하다.

## 2. 리팩토링 제안

### Keep

- Red, Green, 골든마스터 단계 이후 리팩토링 방향을 제안받는 과정에서 AI의 도움을 받을 수 있었다.
- 기존 함수가 여러 가지 책임을 동시에 가지고 있을 때, 하나의 함수가 한 가지 역할만 하도록 분리하는 방향을 제안받을 수 있었다.
- 함수 호출 시 인자를 변경해도 동일한 결과를 유지해야 하는 경우처럼, 리팩토링 후에도 기존 동작이 보존되어야 하는 조건을 확인하는 데 도움이 되었다.

### Problem

- 리팩토링 기준이 명확하지 않으면 AI가 어느 정도까지 코드를 변경해도 되는지 판단하기 어렵다.
- 함수 분리, 인자 변경, 결과 동일성 보장 같은 기준이 구체적으로 정의되지 않으면 리팩토링의 성공 여부를 평가하기 어렵다.
- AI가 제안한 리팩토링이 실제로 기존 동작을 유지하는지 확인하려면 사전에 평가 기준이 더 명확해야 한다.

### Try

- 리팩토링 전에 다음 기준을 명확히 정의한다.
  - 하나의 함수는 하나의 책임만 가진다.
  - 함수 인자가 변경되더라도 동일한 입력 조건에서는 동일한 결과를 반환해야 한다.
  - 기존 UI 동작과 주요 비즈니스 로직은 변경되지 않아야 한다.
  - Red, Green, 골든마스터 테스트 결과를 기준으로 리팩토링 성공 여부를 판단한다.
- AI에게 리팩토링을 요청할 때는 변경 목적, 변경 가능 범위, 변경 금지 영역, 검증 방법을 함께 전달한다.

## 3. 테스트 자동화

### Keep

- 테스트 자동화는 Red, Green, 리팩토링 모든 단계에서 필요하다고 느꼈다.
- AI가 코드 탐색을 잘 수행하면 테스트 대상과 평가 기준을 더 빠르게 도출할 수 있다.
- 골든마스터 테스트를 통해 기존 동작을 기준으로 삼고, 리팩토링 후 결과가 동일한지 확인할 수 있었다.

### Problem

- AI가 평가한 내용이 실제로 정상인지 확인할 수 있는 기준이 부족하면 테스트 자동화의 신뢰도가 낮아진다.
- 코드 탐색 결과가 정확하지 않으면 Red, Green, 리팩토링 단계의 테스트 역시 잘못된 기준으로 작성될 수 있다.
- 테스트가 자동화되어 있더라도 무엇을 통과 기준으로 볼지 명확하지 않으면 결과 해석이 어려워진다.

### Try

- 테스트 자동화 전에 AI의 코드 탐색 결과가 실제 코드와 일치하는지 먼저 확인한다.
- Red 단계에서는 실패해야 하는 조건을 명확히 정의한다.
- Green 단계에서는 최소 구현으로 테스트를 통과하는지 확인한다.
- 리팩토링 단계에서는 기존 테스트와 골든마스터 결과가 유지되는지 확인한다.
- 테스트 결과가 실패할 경우 어느 파일, 어느 함수, 어느 조건에서 실패했는지 추적 가능하도록 로그와 보고 형식을 정리한다.

## 4. 문서화 및 보고서

### Keep

- AI를 활용하면 코드 구조, 테스트 결과, 리팩토링 방향을 문서 형태로 빠르게 정리할 수 있다.
- TDD 단계별 결과를 문서화하면 Red, Green, 리팩토링이 어떤 기준으로 진행되었는지 추적하기 쉽다.
- 문제가 발생한 위치를 코드 파일과 라인 단위로 정리하면 이후 수정과 리뷰가 쉬워진다.

### Problem

- TDD를 시작하기 전에 Spec이 명확하지 않으면 Red, Green, 리팩토링 결과를 객관적으로 판단하기 어렵다.
- Spec이 숫자나 구체적인 기준으로 정의되지 않으면 AI와 사람이 서로 다른 기준으로 성공 여부를 판단할 수 있다.
- 문서에 절차와 검증 기준이 부족하면 리팩토링이 잘 되었는지 나중에 다시 확인하기 어렵다.

### Try

- TDD를 시작하기 전에 Spec이 가능한 한 숫자와 명확한 조건으로 정의되었는지 반드시 확인한다.
- Spec을 기준으로 Red, Green, 리팩토링이 정상적으로 진행되었는지 확인할 수 있는 절차를 문서화한다.
- 보고서에는 다음 내용을 포함한다.
  - 사전 Spec 정의 내용
  - Red 단계에서 실패해야 하는 테스트
  - Green 단계에서 통과한 테스트
  - 리팩토링 후 동일하게 유지되어야 하는 동작
  - 골든마스터 기준과 비교 결과
  - 부족한 부분 또는 실패한 부분
  - 문제가 발생한 코드 파일명과 라인 번호

## 종합 회고

Cursor AI를 활용한 AI Agent 코드 리팩토링은 처음 보는 레거시 코드에 빠르게 진입하고 구조를 이해하는 데 매우 효과적이었다. 특히 클래스 다이어그램, UI와 함수 호출 관계, 시뮬레이션 흐름을 정리해 주는 점은 초기 분석 단계에서 큰 장점이었다.

다만 AI는 프롬프트에 매우 민감하기 때문에, 사람이 보기에는 단순한 실수처럼 보이는 문장도 실제 명령으로 해석할 수 있다. 따라서 단계별 목적과 금지 사항을 명확히 정의해야 하며, 특히 Red, Green, 리팩토링, Git 제출 같은 작업은 서로 분리해서 관리해야 한다.

앞으로는 TDD 시작 전 Spec을 숫자와 조건으로 명확히 정의하고, 이 Spec을 기준으로 테스트 자동화와 리팩토링 검증을 수행하는 방식이 필요하다. 또한 AI가 제안하거나 수행한 결과는 파일명, 함수명, 라인 번호 단위로 기록하여 신뢰할 수 있는 문서와 보고서로 남기는 것이 중요하다.
